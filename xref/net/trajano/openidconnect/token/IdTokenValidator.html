3   /**
4    * <p>
5    * <a href=
6    * "http://openid.net/specs/openid-connect-core-1_0.html#IDTokenValidation">ID
7    * Token validation</a>.
8    * <p>
9    * Clients MUST validate the ID Token in the Token Response in the following
10   * manner:
11   * </p>
12   * <ol>
13   * <li>If the ID Token is encrypted, decrypt it using the keys and algorithms
14   * that the Client specified during Registration that the OP was to use to
15   * encrypt the ID Token. If encryption was negotiated with the OP at
16   * Registration time and the ID Token is not encrypted, the RP SHOULD reject it.
17   * </li>
18   * <li>The Issuer Identifier for the OpenID Provider (which is typically
19   * obtained during Discovery) MUST exactly match the value of the iss (issuer)
20   * Claim.</li>
21   * <li>The Client MUST validate that the aud (audience) Claim contains its
22   * client_id value registered at the Issuer identified by the iss (issuer) Claim
23   * as an audience. The aud (audience) Claim MAY contain an array with more than
24   * one element. The ID Token MUST be rejected if the ID Token does not list the
25   * Client as a valid audience, or if it contains additional audiences not
26   * trusted by the Client.</li>
27   * <li>If the ID Token contains multiple audiences, the Client SHOULD verify
28   * that an azp Claim is present.</li>
29   * <li>If an azp (authorized party) Claim is present, the Client SHOULD verify
30   * that its client_id is the Claim Value.</li>
31   * <li>If the ID Token is received via direct communication between the Client
32   * and the Token Endpoint (which it is in this flow), the TLS server validation
33   * MAY be used to validate the issuer in place of checking the token signature.
34   * The Client MUST validate the signature of all other ID Tokens according to
35   * JWS [JWS] using the algorithm specified in the JWT alg Header Parameter. The
36   * Client MUST use the keys provided by the Issuer.</li>
37   * <li>The alg value SHOULD be the default of RS256 or the algorithm sent by the
38   * Client in the id_token_signed_response_alg parameter during Registration.
39   * </li>
40   * <li>If the JWT alg Header Parameter uses a MAC based algorithm such as HS256,
41   * HS384, or HS512, the octets of the UTF-8 representation of the client_secret
42   * corresponding to the client_id contained in the aud (audience) Claim are used
43   * as the key to validate the signature. For MAC based algorithms, the behavior
44   * is unspecified if the aud is multi-valued or if an azp value is present that
45   * is different than the aud value.</li>
46   * <li>The current time MUST be before the time represented by the exp Claim.
47   * </li>
48   * <li>The iat Claim can be used to reject tokens that were issued too far away
49   * from the current time, limiting the amount of time that nonces need to be
50   * stored to prevent attacks. The acceptable range is Client specific.</li>
51   * <li>If a nonce value was sent in the Authentication Request, a nonce Claim
52   * MUST be present and its value checked to verify that it is the same value as
53   * the one that was sent in the Authentication Request. The Client SHOULD check
54   * the nonce value for replay attacks. The precise method for detecting replay
55   * attacks is Client specific.</li>
56   * <li>If the acr Claim was requested, the Client SHOULD check that the asserted
57   * Claim Value is appropriate. The meaning and processing of acr Claim Values is
58   * out of scope for this specification.</li>
59   * <li>If the auth_time Claim was requested, either through a specific request
60   * for this Claim or by using the max_age parameter, the Client SHOULD check the
61   * auth_time Claim value and request re-authentication if it determines too much
62   * time has elapsed since the last End-User authentication.</li>
63   * </ol>
64   * 
65   * @author Archimedes
66   */
